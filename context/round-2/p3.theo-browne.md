# theo-browne on "PromptRegistry"

**Score:** 7/10
**Would I share this?** yes — killing the "prompt edited out from under the TS app" class of bugs is a real story, and local-file + CI gate is exactly the right architecture

## Strengths

- Dropping the hosted UI removes the main drag from r1. A static manifest file (GitHub raw, R2, S3) that a codegen step turns into `.d.ts` is just tRPC-for-prompts — the mental model already exists in the community and it's good.
- The `promptregistry check` lint step is the killer primitive here. It doesn't require a full codegen re-run, it's just a fast cross-check. That's the kind of CI step you actually wire in because the failure message is legible and the fix is obvious.
- Type contract enforced at build time, not deploy time — this is the sentence. Langfuse and PromptLayer treat prompts as opaque blobs; this makes the shape part of the contract. That's a genuinely different bet and it's the right one for TS shops.

## Concerns

- The codegen story depends on teams actually committing to the manifest-driven discipline. If a PM edits a variable name and nobody runs `promptregistry check` before merging, CI catches it — but only if CI is wired. Most small teams won't wire it until they've been burned. You're shipping the safety net before the falls happen, which means adoption needs to be preceded by education.
- OK so here's the thing: the `pull('prompt-name@v3')` API sounds clean but the version pinning is load-bearing. If teams don't pin versions they get a mutable remote dependency in a typed system. The proposal doesn't say how version bumps surface — that's a footgun waiting to happen and it needs to be explicit in the README before someone builds on this.
- p3 builds on p1 and p2. That's fine for a solo builder but it means the surface area is larger than 4 weeks and the dependency chain makes it harder to evaluate on its own merits. If p1 has a rough edge p3 inherits it silently.

## Pivots

- Make the version pinning story a first-class README section, not an afterthought. Ship a `promptregistry lock` command that writes a lockfile (like package-lock for manifest versions) so the CI hash-check gate is actually checking a pinned hash, not whatever is at the URL today. This one addition turns a potential footgun into the main selling point.
- The launch story should lean into the CI failure screenshot, not the API. Show a PR where a PM removed `{{customer_name}}` and the `tsc --noEmit` step red-lined the build. That screenshot is the hot take. "Prompts with a type contract" doesn't land until people see what the failure looks like.

## Feasibility flags

- **Solo-shippable:** yes — 4-5 weeks given p1 exists as the base; the codegen and check command are the real work
- **Legal/compliance:** clean — static files, no user data, no hosted backend, no compliance surface
- **Stack fit (TS / Svelte / Supabase / AWS / Stripe / OpenAI / Anthropic):** yes — pure TS toolchain, S3/R2 for manifest hosting is optional and already in the builder's stack
- **OSS or paid:** OSS is right for v1; the paid angle (hosted registry, access control, audit logs) is a legitimate round-3 story once CI adoption is proven

## Quote (1 sentence in their voice)

> "OK so here's the thing — a PM removes a variable from a prompt, your TS app silently keeps passing it, and nobody notices until a user gets a broken response in prod; this is the tool that makes that a build error instead."
