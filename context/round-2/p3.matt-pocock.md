# matt-pocock on "PromptRegistry"

**Score:** 8/10
**Would I share this?** yes — a static manifest + `.d.ts` codegen that makes remote prompt edits fail `tsc` is exactly the kind of thing that deserves a blog post: types doing real enforcement work across a network boundary.

## Strengths

- The codegen model is sound. Generating `.d.ts` from a remote manifest is the same pattern as Prisma's schema-to-types or GraphQL codegen — teams already trust it. Applying it to prompts closes a genuine gap nobody else is closing.
- `promptregistry check` as a lint step is the right shape: zero-ceremony, composable, CI-friendly. It doesn't demand a new mental model — you already run `tsc --noEmit`, now you also run `promptregistry check`. That's teachable in one paragraph.
- The scope cut removed every distraction. No Stripe, no Supabase, no hosted UI — what's left is a type system story, which is the only part that's actually interesting.

## Concerns

- The codegen-then-`tsc` discipline will break down the moment someone forgets to re-run codegen. If the `.d.ts` is stale, `tsc` passes while the remote manifest has already drifted — so `promptregistry check` becomes load-bearing, but it's opt-in. That's a silent failure mode that undercuts the whole safety guarantee.
- `pull('prompt-name@v3')` is a string-keyed runtime call. The type safety lives downstream in the generated `.d.ts`, not in the call itself. Advanced TS users will notice that and ask why it isn't a type-level import or a const enum — there's a legitimate "this feels bolted on" critique that needs a good answer in the README.
- Dependency on p1 being shipped and trusted first. If p1 is still rough, p3's `pull()` return type inherits that roughness, and the `.d.ts` story looks shaky. The stack risk is real because it's a three-layer proposal wearing a one-layer jacket.

## Pivots

- Make the stale-codegen problem impossible rather than documented: add a manifest hash to the generated `.d.ts` header and have `promptregistry check` verify it matches the current remote hash, erroring loudly if codegen is stale. That turns "remember to re-run codegen" into a CI-enforced invariant, which is what people actually need.
- Lean into TS module augmentation as a secondary delivery format: alongside the `.d.ts` drop, emit a typed `registry.ts` barrel that re-exports each prompt as a named export (`export { customerSummary } from './prompts/customer-summary'`). Named imports are greppable, refactor-safe, and make the "types doing something genuinely surprising" story obvious to anyone who reads the diff in their PR.

## Feasibility flags

- **Solo-shippable:** yes — 4–5 weeks; the manifest fetch + codegen + lint command is bounded work, and the hard scope cuts eliminate all the surface that would blow the timeline.
- **Legal/compliance:** clean — static JSON fetched from GitHub raw or R2 is no different from downloading an npm package; no PII, no auth, no licensing ambiguity.
- **Stack fit (TS / Svelte / Supabase / AWS / Stripe / OpenAI / Anthropic):** yes — pure TS, zero runtime dependencies outside the p1 primitive; fits the builder's existing stack perfectly.
- **OSS or paid:** OSS is correct here — the value is in the type story going viral, not in gating the manifest format. A paid tier later makes sense only when a hosted registry with write access exists; right now monetisation would just slow adoption.
