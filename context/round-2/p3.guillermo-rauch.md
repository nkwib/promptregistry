# guillermo-rauch on "PromptRegistry"

**Score:** 8/10
**Would I share this?** yes — the static manifest + CI hash gate reframe turns a SaaS liability into a composable primitive with a clear "ship it to your own S3" story.

## Strengths

- Dropping the hosted UI removes the thing that made this feel like a 2022 side project and surfaces the actual primitive: a typed contract between a blob in a bucket and the calling code. That's genuinely new design space.
- The `pull('prompt-name@v3')` surface is the right API shape — version-pinned, explicit, auditable at codegen time. It fits naturally next to how devs already think about package versions and lockfiles. It will read well in a README.
- Static manifest over a public URL (GitHub raw, R2, S3) means zero infra dependency on the author — teams can host their manifest wherever they already store config. That's the kind of zero-lock-in choice that ages well.

## Concerns

- The codegen step is the friction cliff. `tsc --noEmit` failing because a remote URL returned a different shape is a CI surprise most teams won't debug gracefully. The error message has to be exceptional or engineers will just `as any` their way out and the type contract is permanently broken.
- Without a hash gate or integrity check baked into the SDK's `pull()`, someone editing the manifest on a public bucket without bumping the version tag silently invalidates the type contract in running code — the exact failure mode the product is supposed to prevent. The CI hash gate needs to be a first-class SDK feature, not a recipe in the docs.
- Four weeks for a solo build that correctly handles the codegen-to-tsc feedback loop, CLI cross-checking, and S3/R2/GitHub raw fetch normalization is tight. The d.ts emission alone has sharp edges (intersection types, optional variables, partial application). Scope risk is real.

## Pivots

- Ship the hash gate as `promptregistry lock` — a lockfile format (like `prompt-lock.json`, analogous to `package-lock.json`) written by the CLI that pins manifest URL + content hash + version tag together. CI fails if lock is stale. This makes the integrity story self-documenting and gives teams a familiar mental model with zero docs reading required.
- Make the error messages the product. When `tsc --noEmit` catches a shape divergence, the CLI should intercept the TS diagnostic and rewrite it as a human message: "Remote prompt `customer-onboarding@v3` removed variable `customer_name` — update your call site on line 42 or pin to `@v2`." That error message, screenshotted in the README, is the entire marketing pitch.

## Feasibility flags

- **Solo-shippable:** yes — 5 weeks (add one week buffer for d.ts emission edge cases and cross-platform fetch normalization)
- **Legal/compliance:** clean — static manifest over public URLs, no user data handled, no AI provider dependency
- **Stack fit (TS / Svelte / Supabase / AWS / Stripe / OpenAI / Anthropic):** yes — pure TS, S3/R2 are natural manifest hosts, no other stack surface needed
- **OSS or paid:** OSS is correct here — adoption and manifest hosting breadth are the moat; paid only makes sense if a hosted UI ships later and teams want auth + audit logs
