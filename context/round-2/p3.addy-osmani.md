# addy-osmani on "PromptRegistry"

**Score:** 7/10
**Would I share this?** maybe — the static-manifest + codegen pattern is teachable and concrete, but I need to see a before/after adoption story before I'd push it widely.

## Strengths

- The `promptregistry check` lint command is the right shape of tool for how teams actually work. In practice, the linter catches the regression *before* the engineer has to mentally model what codegen did — that's the kind of DX detail that gets real adoption.
- Static manifest over a live API is the correct default for a solo-shippable v1. Fewer moving parts, offline-friendly CI, no secret management overhead for teams who just want a GitHub raw URL. This is pragmatic engineering, not a compromise.
- The separation of concerns is clean: `pull()` + `.d.ts` codegen handles the "remote shape flows into local types" problem, while `promptregistry check` handles the "CI gate without re-running full codegen" problem. These are two distinct user pain points handled by two distinct commands, which is unusual and good.

## Concerns

- The adoption entry point is still steep. Teams have to author and commit a static manifest, wire up a codegen step in CI, and then also wire `promptregistry check` — that's three onboarding steps before anyone sees value. In practice, the first step that breaks or confuses someone is where a tool dies in a team's backlog.
- The dependency on p1 as a peer primitive means this proposal ships no standalone value and inherits all of p1's adoption risk. If p1 doesn't get traction, `pull()` returning a typed callable from that API has no installed base to grow from. That's a real sequencing risk.
- I'd want data on whether PMs or domain experts actually edit raw JSON manifests, or whether "edit prompts without redeploying" in practice means they need a UI anyway. If the manifest is engineer-edited, the no-redeploy pitch is weaker than it looks — you still need a deploy of the manifest file.

## Pivots

- Ship a `promptregistry init` command that scaffolds the static manifest from existing `prompt()` call-sites in the repo via static analysis — zero manual JSON authoring as the entry point. If onboarding is one command instead of three steps, the adoption barrier drops meaningfully.
- Publish a concrete benchmark or case study framing: "before this, a PM removed `{{customer_name}}` from a Langfuse prompt and the app passed it silently for 6 days; after, `promptregistry check` would have caught it in CI in under 2 seconds." That before/after data story is what makes this shareable, not the type theory.

## Feasibility flags

- **Solo-shippable:** yes — 4 weeks is tight but credible if p1 is already stable; the manifest format and codegen are the only genuinely novel surface area
- **Legal/compliance:** clean — static files on GitHub raw / public S3 / R2, no user data, no auth flow
- **Stack fit (TS / Svelte / Supabase / AWS / Stripe / OpenAI / Anthropic):** yes — pure TS CLI + SDK, zero infra in v1, the builder's stack is a direct match
- **OSS or paid:** OSS is correct for v1; the hosted registry + paid tier punt is the right call — prove codegen wires into real CI pipelines before asking for money

## Quote (1 sentence in their voice)

> "In practice, the tools that survive team onboarding are the ones where the first command does something visible — so if `promptregistry init` scaffolds the manifest from existing call-sites automatically, that's the moment this stops being a concept and starts being something a team of five actually ships with."
