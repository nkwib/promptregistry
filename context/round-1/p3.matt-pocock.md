# matt-pocock on "PromptRegistry"

**Score:** 6/10
**Would I share this?** maybe — the codegen-to-types angle is the one genuinely interesting idea here, but the SaaS wrapper buries it under scope that makes this feel like a startup, not a library

## Strengths

- The codegen `.d.ts` flow for pulled prompts is the real gem: you're using code generation to reflect a remote schema back into local types, which is exactly how tRPC and Prisma earned their reputations — this is a teachable, principled pattern
- Pairing semver-style publishing with a typed SDK surface means that breaking changes in a prompt's variable shape are caught at `npm install` time, not at runtime — that's a story I can tell in a workshop
- The problem it solves (PM edits prompt, silently breaks TS caller) is concrete and will resonate with any team that's handed Langfuse access to a non-engineer

## Concerns

- The codegen step is load-bearing but gets one sentence in the MVP scope — if the `.d.ts` generation is flaky, incomplete, or requires manual re-runs, the whole value proposition collapses; this is the riskiest piece and deserves the most attention, not the least
- Six weeks to ship a web editor, semver publishing, a Supabase backend, a Stripe integration, AND a working codegen CLI is wildly optimistic for a solo builder — this is a 3-month project if the types story is done properly
- Competing with Langfuse and PromptLayer on the "hosted prompt management" layer is a distraction; those tools have funding and teams; the differentiation lives entirely in the typed SDK, and that part could ship as a standalone OSS library first

## Pivots

- Invert the build order: ship the OSS SDK + codegen CLI first (call it `prompt-registry-cli`), validate that the `.d.ts` generation actually works reliably across real-world prompt shapes, and only build the hosted UI once engineers are already using the CLI against local or self-hosted manifests — this de-risks the hard part before spending a single hour on Stripe
- Ditch the bespoke web editor for v1 and make the "registry" just a JSON manifest file that can live in a GitHub repo or a public S3 bucket; the CLI pulls from a URL, generates types, done — this gives you a shippable, demonstrable core in two weeks and a genuinely interesting blog post: "Your GitHub repo as a type-safe prompt registry"

## Feasibility flags

- **Solo-shippable:** no — 6 weeks is not enough; the codegen + hosted UI + Stripe combination is realistically 10–12 weeks solo; shippable in 4 weeks only if you drop the hosted UI entirely
- **Legal/compliance:** clean — no notable flags; standard SaaS data handling applies for stored prompts
- **Stack fit (TS / Svelte / Supabase / AWS / Stripe / OpenAI / Anthropic):** yes — matches the builder's stack exactly
- **OSS or paid:** start OSS — the codegen-to-types primitive should be open so the TS community can poke at it; monetise the hosted registry later once the DX is proven
