# addy-osmani on "PromptRegistry"

**Score:** 5/10
**Would I share this?** maybe — the core problem is real and the codegen angle is genuinely novel, but the migration story for teams already on Langfuse or PromptLayer is completely unaddressed, and without that the adoption ceiling is low.

## Strengths

- The type-round-trip idea — pulling a published prompt and generating `.d.ts` so the calling code's types stay in sync with what's live — is the one thing none of the existing prompt-management platforms do. In practice that's the actual failure mode teams hit: a PM edits a prompt in the UI, removes a variable, and the TS app silently keeps passing it. Solving that with codegen is the right level of intervention.
- Scoping to "PM edits, engineer owns calling code" is a credible, narrow user story. Most teams have exactly this split and it causes real breakage. A tight who-for makes the onboarding story crisper than a generic "prompt management" pitch.
- Semver-style versioning on published prompts is the right mental model for a developer audience. Engineers already understand `@v3` as a contract; it sets correct expectations about breaking changes when variable shapes shift.

## Concerns

- The codegen step (`pull('prompt-name@v3')` regenerating `.d.ts`) is the critical path and it's almost entirely unscoped in the MVP. How does it integrate into a CI pipeline? Does it require a `postinstall` hook, a separate CLI command, a pre-commit step? In practice this is where adoption dies — if pulling types requires a manual step that devs forget, the safety guarantee collapses and you're back to an opaque string with extra ceremony.
- The migration story from existing tools is missing. Teams already on Langfuse or PromptLayer have hundreds of prompts stored there. Importing them, mapping variable syntax, and bootstrapping the manifest is a real project, not a nice-to-have. Without a migration path you're only accessible to greenfield projects — a much smaller total addressable team count.
- Six weeks for SDK + Svelte web editor + semver publishing + codegen + Supabase/AWS/Stripe is over-scoped for solo execution. The paid web UI alone is a second product. In practice, shipping a credible codegen story and the OSS SDK to validate the core thesis probably takes the full six weeks by itself.

## Pivots

- Ship only the OSS SDK + CLI codegen in v1: `promptregistry pull` fetches a manifest from a static JSON file (hosted on a GitHub release or a Cloudflare R2 public URL) and writes `.d.ts` locally. No web UI, no Stripe, no Lambda. This validates whether teams actually run the codegen step in CI before you build the paid registry backend. The static-manifest approach also sidesteps auth complexity entirely for the initial signal.
- Add a `promptregistry check` command that reads the local manifest and cross-checks it against all `pull()` call-sites in the repo — flagging mismatched variable names without requiring a full codegen re-run. This gives teams a lint-style safety net that fits into existing CI without changing their build step, which dramatically lowers the adoption bar for the first 100 teams.

## Feasibility flags

- **Solo-shippable:** no — 6 weeks is listed but the full scope (web UI + codegen + backend + Stripe) realistically needs 10–12 weeks solo. The OSS-only slice is shippable in 4–5 weeks.
- **Legal/compliance:** clean — no legal flags; prompts stored server-side are user content, standard SaaS terms apply.
- **Stack fit (TS / Svelte / Supabase / AWS / Stripe / OpenAI / Anthropic):** yes — full stack match; Svelte for the editor, Supabase for prompt storage, AWS Lambda for the pull API, Stripe for billing.
- **OSS or paid:** start OSS-only (SDK + CLI codegen) to prove the codegen adoption loop, then gate the hosted registry behind the paid tier once you have evidence teams run the CLI in CI.

## Quote (1 sentence in their voice)

> "In practice, the codegen step is where this lives or dies — if pulling updated types isn't wired into CI automatically on day one, teams will forget to run it, the type guarantee drifts, and you've built a false sense of safety that's worse than no types at all."
