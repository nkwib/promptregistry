# theo-browne on "PromptRegistry"

**Score:** 5/10
**Would I share this?** maybe — the type-reflecting-back trick is genuinely clever, but I keep bumping into "why isn't this just Langfuse with a codegen step" and I don't have a clean answer

## Strengths

- The core insight — that the deployed prompt's variable shape should flow *back* into the calling code's types via codegen — is actually new. Langfuse and PromptLayer treat prompts as opaque strings at the API boundary. This closes the loop.
- Codegen `.d.ts` per pulled prompt is the right call. It fits exactly how tRPC or Prisma trained TS devs to think: schema-first, types generated, compiler is the test.
- The PM-edits-prompts, engineer-owns-calling-code split is a real pain point that nobody has solved cleanly. That's a real market wedge if you can actually land it.

## Concerns

- OK so here's the thing — you're now asking engineers to add a codegen *and* a deploy step *and* trust a remote registry for a type they depend on. That's three new failure modes before you even run the app. Langfuse exists, works, people already pay for it. What's the 10-second pitch for why I rip that out?
- Semver-style prompt versioning sounds reasonable until a PM publishes `v1.2.0` that removes `{{customer_name}}` and the engineer hasn't re-run codegen yet. You've deferred the problem by one async step — the silent break is still possible, just slower.
- Six weeks for the whole registry + hosted UI + SDK + Stripe is almost certainly underscoped. The SDK alone is p1; the hosted UI is a different product with auth, billing, and editor UX that each take real time. I'd bet this slips to 10+ weeks or ships half-baked.

## Pivots

- Kill the hosted UI for MVP. Ship the SDK + a local-file registry (a `prompts.json` manifest in the repo) and a CLI that generates `.d.ts` from it. Prove the type-round-trip works and gets traction before you build a SaaS. The value prop is the types, not the web editor.
- If you do want a remote registry angle, position this as a Zod-for-prompts-over-the-wire story and add a CI check that fails the build if the pulled prompt's `.d.ts` hash doesn't match what's deployed. That's concrete safety, not vibes — and it's the actual differentiation from Langfuse.

## Feasibility flags

- **Solo-shippable:** yellow — 6 weeks if you cut the hosted UI; 10+ weeks with it. The SDK alone is shippable solo.
- **Legal/compliance:** clean — no flags, prompts stored server-side are just strings
- **Stack fit (TS / Svelte / Supabase / AWS / Stripe / OpenAI / Anthropic):** yes — full stack match
- **OSS or paid:** OSS-first, then paid — the type-safety story needs public adoption before anyone trusts a paid registry with their prod prompts
