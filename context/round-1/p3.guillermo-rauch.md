# guillermo-rauch on "PromptRegistry"

**Score:** 7/10
**Would I share this?** maybe — the registry-as-platform angle is directionally right, but the v1 scope risks becoming another opaque CMS before the type-safe primitive is proven

## Strengths

- The core insight is real: prompt management tools today treat prompts as opaque strings, and that's a primitive-design failure. Reflecting the deployed prompt's variable shape back into the calling code's types is exactly the kind of "what if X were a first-class type?" question worth asking.
- Registry + codegen (`.d.ts` for pulled prompts) is a better design than what Langfuse or PromptLayer ship today. If the SDK surface is tight, this could feel like what Prisma did for databases — the schema becomes the contract.
- PM-edits-without-redeployment is a real workflow problem in AI product teams. That's a genuine pain point, not a manufactured one.

## Concerns

- The registry is useless without trust in the type contract. If a non-engineer edits a prompt in the web UI and breaks the variable shape, does the codegen CLI run in CI before the app deploys? If not, the entire type-safety promise collapses at the integration seam — and that seam is hard to enforce solo.
- Six weeks to ship SDK + web editor + Svelte UI + Supabase + AWS Lambda + Stripe + semver publishing + codegen CLI is not a realistic solo MVP. P1 (promptkit) proves the primitive in 3 weeks; P3 skips proving the primitive and bets on platform before there's adoption.
- The competitive surface is large. Langfuse just shipped prompt versioning with variable tracking. The moment a team like that adds a TS codegen step, this moat compresses fast. What's the compounding platform value that they can't copy in a sprint?

## Pivots

- Ship P1 (the typed `prompt` primitive) first and dogfood it. Then PromptRegistry becomes "we added a registry to the primitive devs already use" rather than "another prompt management SaaS." The SDK moat compounds only if devs are already locked into the tagged-template API.
- Make the CI enforcement first-class in the MVP: a `promptregistry pull && tsc --noEmit` step that fails the build when the remote prompt shape diverges from what the calling code passes. That's the actual value — not the web UI. Defer Stripe until the CI hook has GitHub stars.

## Feasibility flags

- **Solo-shippable:** yellow — 6 weeks is the stated estimate but SDK + codegen + web editor + hosting + payments is realistically 8–10 weeks solo; scope must be cut
- **Legal/compliance:** clean — no data compliance flags, no model provider coupling in the registry layer itself
- **Stack fit (TS / Svelte / Supabase / AWS / Stripe / OpenAI / Anthropic):** yes — exact match to the builder's stack
- **OSS or paid:** hybrid is right, but OSS SDK must ship first and get traction before the paid registry has any pull

## Quote (1 sentence in their voice)

> "A prompt registry that doesn't enforce the type contract at deploy time isn't a registry — it's just a CMS with aspirations."
